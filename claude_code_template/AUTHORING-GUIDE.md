# CLAUDE.md & Rules 작성 가이드

## 핵심 원칙: 토큰 경제학

CLAUDE.md + rules + memory는 **매 턴마다** 전체 로드됨.
200k 컨텍스트 윈도우에서 이들이 차지하는 비율이 곧 "세금".

```
나쁜 예: CLAUDE.md 500줄 + rules 300줄 = 800줄 (~16k 토큰)
→ 매 턴 8% 세금, 100턴 대화 시 1.6M 토큰 낭비

좋은 예: CLAUDE.md 150줄 + rules 200줄 = 350줄 (~7k 토큰)
→ 매 턴 3.5% 세금, 100턴 대화 시 700k 토큰 절약
```

## 무엇을 넣을 것인가 (우선순위)

### Tier 1: 반드시 포함 (가장 높은 ROI)

1. **빌드/테스트/린트 명령어** — Claude가 추측할 수 없는 정보
2. **금지 규칙 ("하지 말 것")** — 모델이 가장 강하게 준수
3. **핵심 패턴 + 참조 파일** — "workinfos.py 패턴을 따를 것"

### Tier 2: 강력 추천

4. **범위 제약** — 원치 않는 변경 방지
5. **응답 스타일** — 간결/상세 제어
6. **에러 처리 패턴** — 일관성 보장

### Tier 3: 선택적

7. **도메인 용어** — 비기술 도메인에서 효과적
8. **Git 워크플로우** — 팀 컨벤션이 있을 때
9. **보안 규칙** — 민감 데이터 다루는 프로젝트

## 무엇을 넣지 말 것인가

| 넣지 말 것 | 이유 | 대안 |
|-----------|------|------|
| 전체 디렉토리 트리 | 코드에서 유추 가능 | Glob으로 탐색 |
| API 엔드포인트 상세 스펙 | 매 턴 불필요 | docs/ 참조 링크 |
| 프레임워크 사용법 | Claude가 이미 앎 | 프로젝트 특화 부분만 |
| Git 히스토리/변경 이력 | git log로 확인 가능 | 필요 시 조회 |
| 전체 코드 예시 | 실제 파일 참조가 나음 | "X.py 패턴 따를 것" |
| 완료된 작업 이력 | 현재에 불필요 | memory로 분리 |
| 자주 변하는 정보 | 관리 부담, 불일치 위험 | memory로 분리 |

## 효과적인 문장 작성법

```markdown
# 나쁜 예 (모호, 검증 불가)
코드를 깨끗하게 작성해주세요.
가능하면 테스트를 추가해주세요.
좋은 변수명을 사용해주세요.

# 좋은 예 (구체적, 검증 가능)
함수는 50줄 이내. Early return 사용.
새 기능 추가 시 해당 기능의 테스트 필수.
boolean 변수: is_, has_, can_ 접두사 사용.
```

## 계층 구조 활용

```
~/.claude/CLAUDE.md                    # 개인 전역 (모든 프로젝트)
  → 응답 언어, 일반 코딩 스타일

프로젝트/CLAUDE.md                     # 프로젝트 공통 (git 커밋)
  → 명령어, 아키텍처, 핵심 패턴

프로젝트/.claude/rules/*.md            # 주제별 규칙 (git 커밋)
  → 테스트, 보안, Git, 에러처리

프로젝트/하위디렉토리/CLAUDE.md        # 하위 디렉토리 (모노레포용)
  → 프론트엔드/백엔드별 규칙

~/.claude/projects/...memory/          # 메모리 (개인, 자동)
  → 사용자 정보, 피드백, 프로젝트 상태
```
