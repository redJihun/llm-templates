## 1. 페르소나 패턴 (Role Assignment)

Claude에게 역할을 부여하면 출력 품질이 측정 가능하게 향상됨.

```markdown
## Role
당신은 프로덕션 칩 제조 시스템을 담당하는 시니어 백엔드 엔지니어입니다.
신뢰성과 정확성을 영리함보다 우선합니다.
불확실할 때는 더 단순한 접근 방식을 선택합니다.
```

**효과**: 도메인 맥락에 맞는 결정, 과도한 추상화 방지

## 2. 행동 전 사고 패턴 (Think Before Acting)

코드 생성 전 맥락 이해를 강제:

```markdown
## 작업 순서
변경하기 전에 반드시:
1. 관련 파일을 먼저 읽는다
2. 기존 패턴을 파악한다
3. 수정할 파일 목록을 언급한다
4. 기존 패턴을 따라 구현한다
5. 린트/테스트로 검증한다
```

**효과**: 맥락 없이 코드 생성하는 문제 방지

## 3. 예시 기반 패턴 (Example-Driven)

추상적 설명 대신 실제 파일을 가리킴 — 매우 효과적:

```markdown
## 패턴 참조
- 새 CRUD 엔드포인트: `routers/workinfos.py` 패턴 따를 것
- 새 ORM 모델: `domain/board/models.py` 패턴 따를 것
- 새 스키마: `domain/board/schemas.py` 패턴 따를 것
```

**효과**: Claude가 참조 파일을 읽고 정확히 복제

## 4. 체크리스트 게이트 패턴 (Checklist Gate)

작업 완료 전 필수 검증:

```markdown
## 작업 완료 전 확인
- [ ] 타입 에러 없음: `ruff check`
- [ ] 테스트 통과: `pytest`
- [ ] console.log / print 제거됨
- [ ] 미사용 import 없음
```

**효과**: 작업 품질 일관성 보장

## 5. 범위 제약 패턴 (Scope Constraint)

가장 중요한 기법 중 하나 — 원치 않는 변경 방지:

```markdown
## 범위 규칙
- 요청받은 파일만 수정한다
- 설정 파일은 사전 확인 없이 수정하지 않는다
- 버그 수정 시 최소 파일만 변경한다
- 요청받지 않은 코드를 리팩토링하지 않는다
- 기존 코드에 docstring/타입 힌트를 추가하지 않는다
```

**효과**: "선의의" 불필요한 변경 방지

## 6. 금지 규칙 패턴 (Negative Constraints)

명시적 금지는 놀라울 정도로 효과적 — 모델이 강하게 준수:

```markdown
## 절대 하지 말 것
- `any` 타입 사용 금지
- 새 파일 생성 금지 (요청 시에만)
- 테스트 fixture 수정 금지
- `git push` 무단 실행 금지
- 이모지 사용 금지
```

**효과**: 반복되는 실수를 영구적으로 차단

## 7. 계층적 컨텍스트 패턴 (Layered Context)

항상 로드되는 파일과 필요 시 읽는 파일을 분리:

```
항상 로드 (CLAUDE.md + rules):  명령어, 핵심 규칙, 제약 조건
                                → 200줄 이내, 최소한으로

필요 시 읽기 (docs/):           API 스펙, 아키텍처 상세, ADR
                                → 무제한, Claude가 필요 시 Read

메모리 (memory/):               사용자 선호, 프로젝트 상태, 피드백
                                → 자동 로드, 간결하게 유지
```

**효과**: 매 턴 토큰 비용 최적화

## 8. 도메인 용어 정의 패턴 (Domain Vocabulary)

비즈니스 용어를 정의하면 오해 방지:

```markdown
## 도메인 용어
- "발급(Issuance)": 칩 제조 및 프로비저닝 프로세스
- "세션(Session)": 칩 프로그래밍 작업 단위 (1배치)
- "작업(Work)": 여러 세션을 포함하는 작업 설정
- "핸들러(Handler)": 칩을 물리적으로 프로그래밍하는 장치
```

**효과**: 비즈니스 용어의 기술적 맥락 혼동 방지

## 적용 우선순위

1. **범위 제약** + **금지 규칙** — 가장 효과적, 모든 프로젝트에 적용
2. **예시 기반** — 기존 코드가 있는 프로젝트에 강력
3. **행동 전 사고** — 복잡한 작업에 필수
4. **체크리스트 게이트** — CI/CD가 없는 환경에서 유용
5. **페르소나** — 도메인 특화 프로젝트에 효과적
6. **계층적 컨텍스트** — CLAUDE.md가 200줄 이상이면 즉시 적용
7. **도메인 용어** — 비기술 도메인(의료, 금융, 제조)에 효과적
